ทีม product กับทีม engineering มักมีแรงดึงคนละทางโดยธรรมชาติ

- ฝั่งหนึ่งอยากปล่อยของเร็ว
- อีกฝั่งอยากให้ระบบนิ่ง

ถ้าไม่มีภาษากลางในการคุยกัน สุดท้ายการตัดสินใจจะกลายเป็นความรู้สึก เช่น "น่าจะโอเค" หรือ "ยังไม่พร้อม" ซึ่งพอระบบโตขึ้น วิธีนี้จะเริ่มใช้ยากมาก

หนึ่งในเครื่องมือที่ช่วยให้สองฝั่งคุยกันบนข้อมูลเดียวกันได้ดีมากคือ **SLO** และ **error budget**

คำพวกนี้ฟังดูเหมือนของทีม infra หรือบริษัท scale ใหญ่ แต่จริง ๆ แล้ว product team ที่กำลังโตยิ่งควรใช้ เพราะมันช่วยตอบคำถามสำคัญว่า:

**เราจะเร็วได้แค่ไหน โดยไม่เผาผลาญความน่าเชื่อถือของระบบเกินไป**

## 1. เริ่มจากแยก SLI, SLO และ SLA ให้ออก

หลายทีมใช้สามคำนี้ปนกันจนสับสน

- **SLI** คือสิ่งที่เราวัด เช่น request success rate, latency, queue processing success
- **SLO** คือเป้าหมายภายใน เช่น 99.9% ของ request ต้องสำเร็จในรอบ 30 วัน
- **SLA** คือข้อตกลงทางธุรกิจกับลูกค้า

ทีม product ส่วนใหญ่ควรเริ่มจาก SLI และ SLO ก่อน ไม่ต้องกระโดดไปทำ SLA ทันที

เป้าหมายของ SLO ไม่ใช่ทำให้ระบบสมบูรณ์แบบ แต่ทำให้ทีมตกลงกันได้ว่า reliability ระดับไหนเหมาะกับงานนี้จริง

## 2. ระบบทุกตัวไม่ต้องมี SLO เท่ากัน

จุดที่หลายทีมพลาดคือพยายามตั้ง SLO สูงมากเท่ากันทุกระบบ

แต่ความจริงคือ:

- หน้า landing page ภายในองค์กร
- ระบบ checkout
- ระบบ sync report ตอนกลางคืน
- recommendation widget

ไม่ได้มี criticality เท่ากัน

ถ้าตั้งเป้าสูงเกินความจำเป็น ทีมจะเสียเวลามากไปกับการ optimize จุดที่ไม่คุ้ม

ถ้าตั้งต่ำเกินไปในเส้นทางสำคัญ ผู้ใช้จะเจ็บโดยตรง

ดังนั้นการตั้ง SLO ควรถามก่อนว่า:

- user คาดหวังอะไรจาก flow นี้
- ถ้ามันพัง 10 นาที ผลกระทบคืออะไร
- latency ที่ช้ากว่าปกติเริ่มกระทบ business เมื่อไร
- ทีมพร้อมลงทุนกับ reliability ระดับไหน

## 3. Error budget คือสิทธิ์ในการเปลี่ยนแปลง

นี่คือแนวคิดที่ผมชอบที่สุด

ถ้าเราตั้ง SLO ไว้ เช่น success rate 99.9% ต่อ 30 วัน นั่นแปลว่าทีมมี "พื้นที่ให้พลาดได้" เล็กน้อย นั่นแหละคือ **error budget**

error budget ไม่ใช่ข้ออ้างให้ระบบพัง แต่เป็นเครื่องมือช่วยบาลานซ์ระหว่าง:

- การปล่อย feature ใหม่
- การทำ migration
- การปรับปรุง reliability

ถ้าระบบยังใช้ error budget ไปน้อย ทีมมีอิสระในการ experiment หรือ deploy งานเสี่ยงมากขึ้นได้

แต่ถ้า error budget ถูกใช้ไปเยอะหรือใกล้หมด นั่นคือสัญญาณว่าทีมควรชะลอ feature บางอย่างแล้วหันมาซ่อมเสถียรภาพก่อน

มันทำให้คำว่า "เร็วหรือเสถียร" ไม่ต้องเป็นสงครามความเห็นอีกต่อไป

## 4. SLO ที่ดีต้องผูกกับ user experience จริง

กับดักที่เจอบ่อยคือวัดสิ่งที่เก็บง่ายแทนสิ่งที่ user รู้สึกจริง

ตัวอย่างเช่น:

- วัด CPU usage แต่ไม่วัด request latency
- วัด service-level success แต่ไม่วัด complete checkout success
- วัดแค่ API 200 response แต่ไม่ดูว่าข้อมูลถูกต้องหรือไม่

คำถามที่ควรถามคือ:

- ผู้ใช้มองว่าฟังก์ชันนี้สำเร็จเมื่อไร
- user journey ไหนคือเส้นทางสำคัญที่สุด
- latency ระดับไหนเริ่มทำให้คนรู้สึกช้า
- failure แบบไหน user ทนได้ และแบบไหนทนไม่ได้

บางที SLO ที่ดีไม่ได้อยู่ระดับ service เดี่ยว แต่อยู่ระดับ journey เช่น:

- สมัครสมาชิกสำเร็จ
- จ่ายเงินสำเร็จ
- อัปโหลดไฟล์สำเร็จ
- ค้นหาแล้วได้ผลลัพธ์ในเวลาที่ยอมรับได้

## 5. Product team ควรใช้ SLO ตอนวาง roadmap ด้วย

หลายทีมใช้ SLO แค่ตอน incident ซึ่งเสียดายมาก

จริง ๆ แล้ว SLO ควรมีผลกับ roadmap เช่น:

- ถ้า error budget ใกล้หมด เราควรเลื่อน feature บางตัวไหม
- ถ้า flow สำคัญ latency สูงขึ้นเรื่อย ๆ ควรลงทุนแก้ infra หรือ query optimization เมื่อไร
- ถ้าระบบมี reliability ดีกว่าเป้ามาก เราควรเร่ง experiment ได้ไหม

นี่ทำให้ SLO กลายเป็นส่วนหนึ่งของ product strategy ไม่ใช่แค่ dashboard ของ infra team

## 6. ระวัง SLO ที่วัดเยอะแต่ไม่มีใครใช้

อีกฝั่งหนึ่งคือบางทีมตั้ง SLO เยอะมากจนไม่มีใครจำได้ว่าตัวไหนสำคัญ

ผลคือ:

- alert เยอะ
- dashboard เต็ม
- retrospective สับสน
- เวลาตัดสินใจจริงไม่มีใครหยิบตัวเลขไหนมาใช้

วิธีที่ practical กว่าคือเริ่มจากเส้นทางสำคัญ 1-3 อย่างก่อน แล้วตั้ง SLI/SLO ที่ทีมใช้คุยกันจริง

เช่นสำหรับ product หนึ่งตัว อาจเริ่มจาก:

- login success rate
- checkout success rate
- p95 latency ของ search

เมื่อทีมเริ่มใช้เป็น ค่อยขยาย

## 7. วิธีเริ่มโดยไม่ทำให้ทีมรู้สึกว่าซับซ้อนเกินไป

ถ้าทีมยังไม่เคยใช้เลย ผมแนะนำ flow นี้:

1. เลือก user journey สำคัญที่สุด 1 อย่าง
2. นิยามว่า success ของ journey นั้นคืออะไร
3. เลือก metric ที่สะท้อนประสบการณ์จริง
4. ตั้งเป้าที่ "ดีพอ" ไม่ใช่ "สมบูรณ์แบบ"
5. review ทุกเดือนว่าตัวเลขช่วยตัดสินใจอะไรได้บ้าง

จุดสำคัญคืออย่าเริ่มจากเอกสารยาวหรือ taxonomy ใหญ่โต เริ่มจากบทสนทนาที่ทำให้ product และ engineering เห็นภาพเดียวกันก่อน

## 8. Error budget จะมีค่าก็ต่อเมื่อมันเปลี่ยนพฤติกรรมได้

หลายองค์กรมี error budget แต่ไม่มีผลกับการตัดสินใจจริง สุดท้ายมันก็เป็นแค่กราฟอีกเส้น

สิ่งที่ควรถามให้ชัด:

- ถ้า budget หมด เราจะทำอะไร
- ใครตัดสินใจหยุด release หรือชะลอ rollout
- ต้องมีเงื่อนไขอะไรถึงจะกลับมาเร่ง feature ได้
- incident review จะเชื่อมกับ budget อย่างไร

ถ้าตอบคำถามเหล่านี้ไม่ได้ แปลว่าทีมยังมี SLO แต่ยังไม่มี operating model ที่รองรับมัน

## สรุป

SLO และ error budget ไม่ได้มีไว้ทำให้ทีมกลัวการเปลี่ยนแปลง ตรงกันข้าม มันมีไว้เพื่อให้ทีมเปลี่ยนแปลงได้เร็วขึ้นแบบมีขอบเขตที่เข้าใจร่วมกัน

เมื่อ product กับ engineering มีภาษากลางเรื่อง reliability เราจะคุยกันได้ง่ายขึ้นว่าเมื่อไรควรเร่ง เมื่อไรควรชะลอ และเมื่อไรควรลงทุนแก้รากของปัญหา

สุดท้ายแล้วระบบที่ดีไม่ใช่ระบบที่ไม่เคยพลาดเลย แต่เป็นระบบที่ทีม **รู้ว่ารับความเสี่ยงได้แค่ไหน และตัดสินใจได้อย่างมีสติเมื่อเข้าใกล้เส้นนั้น** ครับ
