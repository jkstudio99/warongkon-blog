สวัสดีครับ! หัวข้อ "Dapper vs EF Core" เป็นเรื่องที่ถกเถียงกันมาเป็นสิบปีแล้วนะครับ และในปี 2026 นี้ แม้ทั้งคู่จะพัฒนาไปไกลมาก แต่คำถามที่ว่า "ตัวไหนดีกว่ากัน?" ก็ยังคงมีมาเรื่อยๆ

วันนี้ผมในฐานะ Senior Developer ที่ต้องดูแลระบบหลังบ้านขนาดใหญ่ จะมาสรุปให้ฟังแบบชัดๆ ว่าในยุค .NET 10 นี้ แต่ละตัวมีจุดยืนตรงไหน และเราควรเลือกใช้อย่างไรให้เหมาะสมกับงานครับ

## 🦄 EF Core 10: ความเก่งกาจที่มาพร้อมความสะดวก

ในปี 2026 **EF Core** (Entity Framework Core) ไม่ใช่อืดอาดเหมือนสมัยก่อนอีกแล้วครับ มันได้รับการปรับปรุงประสิทธิภาพจนเรียกได้ว่า "เร็วพอ" สำหรับ 90% ของงานทั่วไป

### ✅ จุดเด่น:
- **Productivity:** เขียนโค้ดน้อยลงมาก ระบบ Migration จัดการเรื่อง Schema ให้หมด
- **LINQ:** ช่วยให้เราเขียน Query ได้ด้วย C# ซึ่งอ่านง่ายและ Type-safe
- **Change Tracking:** จัดการเรื่องการ Update ข้อมูลได้ซับซ้อนและแม่นยำ
- **AOT Support:** รองรับการ Compile แบบ Native AOT แล้ว ทำให้รันเร็วตั้งแต่ออกตัว

### ❌ จุดด้อย:
- **Complexity:** ถ้าไม่เข้าใจพฤติกรรมของมัน อาจจะทำให้เกิดปัญหาประสิทธิภาพ (เช่น N+1 Query) ได้ง่าย
- **Overhead:** ยังคงมีการกินทรัพยากรมากกว่า Micro-ORM เล็กน้อย

---

## ⚡ Dapper: ราชาแห่งความเร็วและความคุมได้

**Dapper** ยังคงครองใจนักพัฒนาที่หลงรักใน SQL และต้องการการควบคุมสูงสุด

### ✅ จุดเด่น:
- **Speed:** ประสิทธิภาพใกล้เคียงกับ ADO.NET แบบดิบๆ มากที่สุด
- **SQL Control:** เราเขียน SQL เอง 100% ทำให้ปรับจูน (Tune) ได้ทุกจุด
- **Simplicity:** เข้าใจง่าย ไม่มีลูกเล่นซ่อนเร้น (No magic)
- **Lightweight:** แทบไม่เพิ่มภาระให้กับแอปพลิเคชัน

### ❌ จุดด้อย:
- **Manual Work:** ต้องเขียน SQL เองทั้งหมด รวมถึงการจัดการ Schema และ Mapping ด้วยมือ
- **No LINQ:** เสี่ยงต่อการพิมพ์ SQL ผิด (แต่มี Tool ช่วยเช็คได้ในปี 2026)

## 🛠️ สูตรสำเร็จปี 2026: ใช้ทั้งคู่ (The Hybrid Approach)

จากประสบการณ์ของผม วิธีที่ดีที่สุดในโปรเจกต์ Enterprise คือ **"ใช้ร่วมกัน"** ครับ!

1. **ใช้ EF Core สำหรับงาน CRUD ทั่วไป:** งานเพิ่ม ลบ แก้ไข ข้อมูลพื้นฐานที่มีความสัมพันธ์ไม่ซับซ้อนมาก EF Core จะช่วยให้คุณส่งงานได้เร็วและโค้ดสะอาด
2. **ใช้ Dapper สำหรับงาน Read หนักๆ หรือ Report ซับซ้อน:** เมื่อต้อง Join ตารางเป็นสิบๆ ตัว หรือต้องการดึงข้อมูลปริมาณมหาศาลเพื่อทำ Report การเขียน SQL ดิบผ่าน Dapper จะให้ความเร็วที่เหนือกว่าอย่างเห็นได้ชัด

## 💡 สรุปมุมมองระดับ Senior

อย่าหลงรักเครื่องมือจนเกินไปครับ ในฐานะนักพัฒนา เราควรเลือกเครื่องมือให้เหมาะกับปัญหา:
- ถ้าต้องการ **Speed of Development** -> เลือก **EF Core**
- ถ้าต้องการ **Speed of Execution** -> เลือก **Dapper**

ในโปรเจกต์ใหญ่ๆ ของผม มักจะเริ่มด้วย EF Core ก่อนเสมอ และจะเปลี่ยนเฉพาะบางส่วนที่พิสูจน์แล้วว่าเป็น Bottleneck ให้เป็น Dapper ครับ

หวังว่าบทความนี้จะช่วยให้เพื่อนๆ ตัดสินใจเลือก ORM ได้มั่นใจขึ้นนะครับ! ใครมีเทคนิคเด็ดๆ ในการใช้งาน มาแชร์กันได้ในคอมเมนต์ครับ
