ปี 2026 หลายองค์กรเริ่มจริงจังกับคำว่า **Engineering Productivity** มากขึ้น เพราะทุกทีมถูกคาดหวังให้ส่งงานเร็วขึ้น ใช้ AI tool ได้ดีขึ้น และควบคุมต้นทุนได้ดีขึ้นพร้อมกัน

แต่ปัญหาคือคำว่า productivity ในทีม developer วัดพลาดง่ายมาก ถ้าวัดแบบผิวเผินมันจะกลายเป็น scoreboard ที่ทำลาย trust แทนที่จะช่วยให้ทีมทำงานดีขึ้น

บทความนี้ชวนมองว่าเราควรวัดอะไร วัดไปเพื่ออะไร และจะใช้ metric ยังไงให้ทีมรู้สึกว่า data เป็นตัวช่วย ไม่ใช่กล้องจับผิด

## 1. อย่าเริ่มจากคำถามว่าใครทำงานเยอะที่สุด

กับดักแรกคือการเอา metric ไปวัดรายบุคคลเร็วเกินไป เช่น นับจำนวน commit, จำนวน PR, จำนวนบรรทัดโค้ด หรือจำนวน ticket ที่ปิดได้

ตัวเลขพวกนี้ดูง่าย แต่หลอกง่ายมากเช่นกัน

developer ที่แก้ incident ยากหนึ่งเคสอาจมี commit น้อยกว่า developer ที่แยก formatting เป็นสิบ commit แต่ value ที่สร้างไม่เท่ากันเลย

คำถามที่ดีกว่าคือ:

- งานติดตรงไหนบ่อยที่สุด
- feedback loop ช้าตรงไหน
- handoff ระหว่างทีมทำให้เสียเวลาแค่ไหน
- release แล้วเกิดปัญหาซ้ำจากจุดไหน
- developer เสียเวลาไปกับงานที่ไม่ใช่ product เท่าไร

ถ้าเราเริ่มจากระบบงาน ไม่ใช่เริ่มจากจับผิดคน metric จะมีโอกาสช่วยทีมได้จริงมากกว่า

## 2. Flow metrics ช่วยให้เห็นคอขวดของระบบ

หนึ่งในชุด metric ที่มีประโยชน์มากคือ **flow metrics** เพราะมันไม่ได้ถามว่าใครยุ่งที่สุด แต่ถามว่างานไหลผ่านระบบได้ดีแค่ไหน

ตัวอย่างที่ใช้ได้จริง:

- Lead time: ตั้งแต่เริ่มงานจนถึง production ใช้เวลานานแค่ไหน
- Cycle time: ตั้งแต่เริ่ม implement จน merge ใช้เวลานานเท่าไร
- Review time: PR รอ review นานแค่ไหน
- Deployment frequency: deploy บ่อยและมั่นคงแค่ไหน
- Blocked time: งานติดสถานะ blocked นานแค่ไหน

ถ้าเห็นว่า PR ส่วนใหญ่เขียนเสร็จเร็วแต่รอ review 2 วัน ปัญหาไม่ได้อยู่ที่ dev เขียนโค้ดช้า แต่อยู่ที่ review capacity หรือ process

ถ้าเห็นว่า merge แล้ว deploy ช้า ปัญหาอาจอยู่ที่ pipeline, release policy หรือ confidence ใน test suite

metric ที่ดีควรพาทีมไปคุยเรื่องระบบ ไม่ใช่พาไปโทษคน

## 3. DORA ยังใช้ได้ แต่ต้องอ่านให้เป็น

DORA metrics ยังเป็น baseline ที่ดีสำหรับทีม software:

- Deployment frequency
- Lead time for changes
- Change failure rate
- Mean time to recovery

แต่ข้อควรระวังคืออย่าใช้มันแบบ copy-paste โดยไม่ดูบริบท

ทีมที่ทำ internal admin tool กับทีมที่ดูแล payment system อาจมี acceptable risk ไม่เท่ากัน ทีมที่อยู่ในช่วง rewrite หรือ migration ใหญ่ก็อาจมีตัวเลขแกว่งกว่า team ที่ product mature แล้ว

สิ่งสำคัญคือใช้ DORA เพื่อถามว่า "เรากำลังดีขึ้นไหม" มากกว่าใช้เพื่อถามว่า "เราดีกว่าทีมอื่นไหม"

## 4. Developer Experience เป็น metric ที่ควรถามตรง ๆ

หลายอย่างที่กระทบ productivity ไม่ได้เห็นจาก git หรือ CI เสมอไป เช่น:

- local dev setup เจ็บปวดไหม
- test flaky จนคนไม่เชื่อหรือเปล่า
- เอกสารหาไม่เจอหรือเก่าเกินไปไหม
- dev ต้องรอ permission หรือ environment บ่อยแค่ไหน
- tooling ทำให้เสียสมาธิหรือช่วยลดงานซ้ำจริง

เรื่องพวกนี้วัดได้จาก survey สั้น ๆ หรือ retro ที่มีคำถามชัด

ตัวอย่างคำถามที่ดี:

- สัปดาห์นี้อะไรทำให้คุณส่งงานช้าลงมากที่สุด
- ถ้าลบ friction ได้หนึ่งอย่าง คุณอยากลบอะไร
- เครื่องมือไหนช่วยคุณจริง และเครื่องมือไหนเพิ่มภาระ
- คุณมั่นใจแค่ไหนว่า deploy แล้วจะรู้เร็วถ้ามีปัญหา

บางครั้งคำตอบจาก developer 5 คนมีค่ากว่า dashboard 10 หน้า เพราะมันบอก pain point ที่ตัวเลขยังอธิบายไม่หมด

## 5. AI productivity ต้องวัด outcome ไม่ใช่ token

เมื่อ AI coding tool และ agent เข้ามาใน workflow ทีมเริ่มอยากรู้ว่ามันช่วยได้จริงไหม

สิ่งที่ไม่ควรวัดเป็นหลักคือจำนวน prompt, token usage หรือจำนวนโค้ดที่ AI generate เพราะตัวเลขพวกนี้ไม่ได้บอกว่า software ดีขึ้นหรือแย่ลง

สิ่งที่ควรวัดมากกว่า:

- task ประเภทไหนเร็วขึ้นจริง
- bug จาก AI-generated code เพิ่มหรือไม่
- review burden เพิ่มหรือลด
- test coverage ของงานที่ใช้ AI เป็นอย่างไร
- developer ใช้ AI แล้วเข้าใจโค้ดที่ส่งไหม

AI ที่ดีไม่ควรแค่ทำให้เราเขียนโค้ดเร็วขึ้น แต่ควรทำให้ทีมได้ feedback เร็วขึ้นและลดงานซ้ำที่น่าเบื่อได้จริง

## 6. ระวัง metric ที่สร้างพฤติกรรมผิด

ทุก metric มีแรงจูงใจซ่อนอยู่

ถ้าวัดจำนวน PR คนจะแตก PR เล็กเกินจำเป็น

ถ้าวัดจำนวน ticket ที่ปิด คนจะเลี่ยงงานยาก

ถ้าวัด velocity แบบแข็งเกินไป ทีมจะ inflate estimate

ถ้าวัด deploy frequency โดยไม่ดู failure rate ทีมอาจ deploy บ่อยแต่คุณภาพตก

หลักง่าย ๆ คือ ถ้า metric ใดทำให้คนต้องเล่นเกมกับตัวเลข แปลว่า metric นั้นเริ่มกลายเป็นปัญหาแล้ว

## 7. วิธีเริ่มแบบไม่ทำให้ทีมระแวง

ถ้าองค์กรยังไม่เคยวัด engineering productivity จริงจัง ผมแนะนำให้เริ่มเล็กและโปร่งใส

1. บอกให้ชัดว่าจะใช้ metric เพื่อปรับระบบงาน ไม่ใช่ประเมินรายบุคคล
2. เริ่มจาก team-level metric ก่อน
3. เลือก 3-5 ตัวที่ทีมเข้าใจและเชื่อมกับ pain point จริง
4. เอาข้อมูลเข้าวง retro เพื่อหา improvement
5. ทบทวน metric ทุก 1-2 เดือนว่ามันยังช่วยอยู่ไหม

ทีมจะยอมรับ data ได้ง่ายขึ้นเมื่อเห็นว่า data นำไปสู่การแก้ friction จริง เช่น ลดเวลารอ review, แก้ flaky test, ปรับ deploy pipeline หรือทำ template ให้เริ่มงานง่ายขึ้น

## สรุป

Engineering productivity ที่ดีไม่ได้เริ่มจาก dashboard สวย ๆ แต่เริ่มจากความตั้งใจที่ถูกต้อง: **เราจะทำให้ระบบงานช่วยคนทำงานได้ดีขึ้นอย่างไร**

วัด flow เพื่อหาคอขวด วัด DORA เพื่อดู health ของ delivery วัด developer experience เพื่อฟังสิ่งที่ dashboard มองไม่เห็น และวัด AI ด้วย outcome ไม่ใช่ความตื่นเต้น

ถ้าทีมใช้ metric เพื่อเรียนรู้ร่วมกัน มันจะกลายเป็นเครื่องมือที่ดีมาก แต่ถ้าใช้เพื่อจับผิด มันจะทำให้คนเก่ง ๆ หยุดพูดความจริง และนั่นแพงกว่าตัวเลข productivity ที่ดูสวยมากครับ
